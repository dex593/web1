# Button UX Skill Rules (Mandatory)

This file is a strict policy for all future code touching buttons in this repo.

## 1) Scope
- Applies to all routes and all UI entry points.
- Applies to both:
  - static buttons in `views/**/*.ejs`
  - dynamic buttons created in `public/*.js`

## 2) Icon policy
- Every action button MUST have a meaningful icon.
- Do NOT add icon twice:
  - if button already has `fa-*`, `<svg>`, or `<img>`, keep as is.
  - only add icon when missing.
- Prefer context mapping over generic mapping.
- For new button types, add stable selectors first (`data-*` or class), then map in:
  - `public/button-ui.js` -> `resolveIconFromAttributes(...)`
- If needed, extend:
  - `exactTextIconMap`
  - `keywordRules`

## 3) Required no-icon exceptions
- Genre filter toggle buttons MUST NOT have icon.
- Keep these as no-icon classes (or equivalent explicit skip):
  - `filter-option`
- For non-action list/select items, skip auto icon via:
  - `data-button-icon-skip`
  - or add class to `ICON_SKIP_CLASSES` in `public/button-ui.js`

## 4) Loading/processing state policy
- Any async button action MUST show pending state until done.
- Pending state behavior:
  - button is dimmed
  - spinner is shown
  - interaction is blocked (disabled or equivalent)
- Standard mechanism is already implemented globally:
  - JS: `public/button-ui.js`
  - CSS: `public/styles.source.css` (`button.is-loading`)
- For submit buttons: rely on built-in auto handling.
- For click-only async buttons:
  - set `button.disabled = true` before request
  - restore in `finally`
  - OR call `window.BfangButtonUi.setLoading(button, true/false)`

## 5) Accessibility rules
- Icon-only buttons MUST keep `aria-label`.
- Decorative icons MUST use `aria-hidden="true"`.
- Do not remove visible labels unless there is a clear UX reason.

## 6) Integration rules
- `button-ui.js` must stay globally loaded from `views/partials/head.ejs`.
- Do not disable button UI script per page unless there is a documented reason.

## 7) Definition of done for button-related PRs
- All newly added action buttons have correct icon semantics.
- No duplicate icons on existing icon buttons.
- Async actions show loading state from start to completion/failure.
- At least one pass verifies:
  - static buttons in affected EJS files
  - dynamic buttons generated by affected JS files
- Syntax checks pass for touched JS files (`node --check ...`).

## 8) Priority order for mapping decisions
1. `data-*` specific mapping (most reliable)
2. class-specific mapping
3. exact text mapping
4. keyword mapping
5. generic fallback (only for true action buttons)

## 9) Notification policy (toast-only)
- All user-facing operation feedback (success/error/warning/info) MUST be shown as toast at the bottom of the viewport.
- Inline status blocks are fallback only; if they are used by legacy code, they must be bridged to toast behavior.
- Toasts MUST include:
  - semantic type (save/delete/create/update/approve/reject/copy/send/auth/upload/search/error/warning/info)
  - distinct color style per type
  - matching Font Awesome icon
- For async actions, emit final toasts for both success and failure outcomes.
- Avoid blocking browser alerts for normal UX; use toast instead.
- Keep dedupe so the same message is not spammed repeatedly.

Treat these rules as mandatory, not optional guidance.
